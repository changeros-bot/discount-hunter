# V17 Quality Gate 系統架構規範

## 核心定位

Quality Gate 是 DCA 與逢低買進策略的前置斷路器（Circuit Breaker）。

它不決定買入價格；買入價格由 Discount Hunter 的 D 層規則決定。Quality Gate 只決定該標的是否具備「分配資金的權限」。

核心目的：避免用純技術面或 FOMO 心態，持續攤平一家基本面已經破壞的公司或不合格資產。

## 一、能不能買 vs 怎麼買

| 問題 | 負責模組 | 說明 |
|---|---|---|
| 能不能買 | Quality Gate | 決定標的是否合格，是否能分配資金 |
| 怎麼買 | Discount Hunter D 層 | 決定價格、層級、建議金額 |
| 是否執行 | 現金檢查 + 半自動確認 | 決定當下是否真的下單 |

## 二、三層稽核架構

### 第一層：客觀財務，滿分 10 分

適用於一般股票與 xStocks。xStocks 查底層股票，例如 NVDAon 查 NVIDIA。

每項評分：通過 2 分，觀察 1 分，失敗 0 分。

| 稽核項目 | 具體審查核心 |
|---|---|
| 營收成長 | 最近年度 / TTM 是否仍維持增長軌道 |
| 自由現金流 | FCF 是否穩定轉正，是否具備健康自我造血能力 |
| 毛利率趨勢 | 毛利率是否維持產業競爭力，有無異常惡化或定價權喪失 |
| 資產負債表 | 現金儲備、整體負債結構、利息覆蓋率與償債壓力 |
| 資本支出趨勢 | CapEx 投資是否合理且具備未來投報率，而非惡性燒錢 |

詳細數字門檻見：`docs/V17_QUALITY_GATE_NUMERIC_RULES.md`。

### 第二層：質化條件，滿分 8 分

驗證「當初納入折扣獵人的核心邏輯是否依然成立」。

每項評分：通過 2 分，觀察 1 分，失敗 0 分。每項必須寫一句理由，不允許只給分數。

| 稽核項目 | 具體審查核心 |
|---|---|
| 產業領導地位 | 是否仍處於產業核心節點，具備龍頭或關鍵供應商優勢 |
| 經濟護城河 | 技術壁壘、品牌溢價、規模效應或客戶黏性是否受損 |
| 管理層品質 | 財測執行力是否穩定，資本配置是否合理 |
| 投資假設驗證 | 當初買入這檔標的的底層故事，有沒有被市場或競爭對手破壞 |

理由格式：

> 護城河：通過 — CUDA 生態系與開發者黏性仍在，短期未見有效替代。

### 第三層：交易權限判定，股票 / xStocks 滿分 18 分

| 總分區間 | 判定結果 | 交易權限定義 |
|---|---|---|
| 14–18 分 | 通過 | 且無重大失敗項。可固定 DCA、可逢低、允許半自動草稿、可列自動交易白名單候選 |
| 9–13 分 | 觀察 | 可繼續持有，固定 DCA 改小額或低優先，逢低買進降為低優先級 |
| 資料不足 | 未檢查 | 無法完整稽核。不進自動交易，依賴人工確認 |
| 8 分以下 | 失敗 | 啟動斷路器：停止新增資金，不產生草稿，不進自動交易 |

## 三、一票否決制

以下重大失敗項不能被總分掩蓋：

| 重大項目 | 處理 |
|---|---|
| 自由現金流 FCF 失敗 | 最高只能觀察；若同時營收衰退，降為失敗 |
| 資產負債表失敗 | 最高只能觀察；若有流動性危機，直接失敗 |
| 投資假設破壞 | 直接失敗 |

## 四、非標準標的特殊稽核框架

| 標的類型 | 稽核框架 |
|---|---|
| 股票 / xStocks | 標準財報五項 + 質化四項 |
| QQQon / ETF | 指數代表性、成分集中度、長期成長邏輯、費用率、流動性 |
| BTC | 網路安全、流動性、採用趨勢、監管風險、託管/交易所風險 |
| SPCXon | 新上市 / 交易歷史不足。使用上市以來高點，逢低人工確認 |

### SPCXon 修正版

SPCXon 不再使用「非上市公司」處理。2026-07-08 系統內定位為：新上市 / IPO 後交易歷史不足 / 財報與歷史高點資料仍待完整驗證。

| 項目 | 規則 |
|---|---|
| 52 週資料 | 未滿 52 週，不得使用完整 52 週高點框架 |
| 高點基準 | 使用上市以來高點 |
| Quality 狀態 | 預設未檢查或觀察，不得直接通過 |
| DCA | 可保留 5U 小額 DCA |
| 逢低 | 必須人工確認資料源與上市以來高點 |
| 自動交易 | 不進自動交易白名單，直到至少兩季財報與足夠交易歷史 |

## 五、稽核結果對策略執行的映射

| Quality 狀態 | 固定 DCA 權限 | 逢低買進權限 | 半自動草稿 | 未來自動交易白名單 |
|---|---|---|---|---|
| 通過 | 正常執行 | 正常執行 | 自動生成草稿 | 准入候選 |
| 觀察 | 降低配置或低優先 | 標記低優先 | 需手動確認 | 移出白名單 |
| 未檢查 | 僅保留既定小額 | 禁止自動執行 | 全人工確認 | 移出白名單 |
| 失敗 | 徹底封鎖 | 徹底封鎖 | 徹底封鎖 | 徹底封鎖 |

## 六、既有部位處置

| Quality 變化 | 既有部位 | 新增資金 |
|---|---|---|
| 通過 → 觀察 | 不主動減碼，保留持倉 | 停止高優先逢低，只允許低優先或人工確認 |
| 觀察 → 失敗 | 進入退出/減碼檢查，不自動賣出 | 停止 DCA、逢低與草稿 |
| 未檢查 | 保留既有小額觀察 | 不進自動交易，逢低需人工確認 |
| 通過維持通過 | 持續原策略 | 可 DCA、可逢低、可草稿 |

## 七、UI 展現規範

### App 主卡片

主卡片只顯示結論，不塞完整稽核內容。

範例：

```text
NVDAon
Quality：通過
DCA：允許
逢低：允許
半自動：可草稿
```

完整稽核內容放在 `/v17-quality` 或獨立報告頁。

## 九、自動化資料策略

本專案以 Next.js / Vercel 為主，不以 Google Apps Script 作為核心執行環境。

| 層級 | 自動化策略 |
|---|---|
| 第一層客觀財務 | V17.4 接財務資料 API，定期更新初篩分數 |
| 第二層質化條件 | 每季或重大事件由 Josh 人工確認 |
| 權限映射 | 由 Quality Gate API 輸出給半自動草稿與未來自動交易模組 |

## 十、Asset Admission Process 銜接

Quality Gate 是每次階段升級的必要條件，不是只在最初納入時做一次。

| 階段 | Quality 要求 | 權限 |
|---|---|---|
| Research | 可未檢查 | 只研究，不交易 |
| Watchlist | 至少完成初審 | 觀察，不買或只提醒 |
| Paper Tracking | 至少觀察 | 紙上追蹤 |
| Small Position | 觀察以上 | 小額人工買入 |
| Core Candidate | 通過且無重大失敗項 | 可 DCA / 可逢低 / 可草稿 |
| Core Asset | 連續兩次季度通過 | 可列自動交易白名單候選 |
